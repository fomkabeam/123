# Чек-лист проверки нововведений

Порядок действий для проверки всех последних изменений (Clang-пайплайны, GUI, граф вызовов).

---

## 1. Подготовка конфигурации

1. Откройте **LOG_PATH.txt** в корне проекта `С++`.
2. Убедитесь, что заданы:
   - **DB_NAME, DB_USER, DB_PASS, DB_HOST, DB_PORT** — доступ к PostgreSQL.
   - **PATH_FILE** — полный путь к каталогу с исходниками C++ (корень проекта).
   - **SEARH_METH=CLANG_AST** (или **CLANG** с диагностикой, или **CLANG_AST_NO_SENS**).
   - **COMPILE_COMMANDS** — полный путь к `compile_commands.json` (из сборки проекта с `CMAKE_EXPORT_COMPILE_COMMANDS=ON`).
3. Сохраните файл.

---

## 2. Запуск анализа (_Starter)

1. Из каталога `С++` выполните:
   ```bash
   python _Starter.py
   ```
   (или запустите через GUI — кнопка «Запуск анализа» / старт из главного окна.)
2. Дождитесь окончания без ошибок.
3. Ожидаемый результат:
   - В консоли — последовательный запуск шагов пайплайна (find_files, clang_ast_runner, clang_ast_variables и т.д.).
   - В БД заполнены: **ccc_file_list**, **ccc_definition_function**, **ccc_use_function_list**, **ccc_definition_variable**, **ccc_use_variable_list**, представление **ccc_not_use_variable_ast**.

---

## 3. Проверка GUI — результаты

1. Запустите GUI (например `python gui_analyzer.py` или через главный лаунчер).
2. Откройте раздел **«Результаты»** (или вкладку с таблицами по функциям/переменным).
3. Проверьте:

   **Неиспользуемые переменные (AST):**
   - Должна быть таблица с колонками: **ID**, **ID файла**, **Путь**, **Имя**, **Строка**, **Код**, **Область**, **Критичность**, **Инициализ.**, **Безопасность**.
   - Строки отсортированы по критичности (CRITICAL → HIGH → MEDIUM → LOW).

   **Все переменные:**
   - Таблица из **ccc_definition_variable** (колонки: ID переменной, ID файла, Имя, Тип, Строка), либо fallback на **ccc_variables_v**, если первой нет.

4. При пустых таблицах — убедитесь, что анализ (шаг 2) выполнен с **SEARH_METH=CLANG_AST** (или CLANG) и что в проекте есть объявления переменных/функций.

---

## 4. Проверка построения графа вызовов

1. В GUI откройте окно **«Построение графа вызовов»**.
2. Укажите:
   - **Каталог с исходниками C++** — тот же путь, что и **PATH_FILE** (или каталог, для которого нужен граф).
   - **Каталог для сохранения графа** — например `GraphAnalyserC++/graphs` или свой путь.
3. Нажмите **«Построить граф вызовов»**.
4. Дождитесь сообщения «Граф сохранён в каталог: ...».
5. В указанном каталоге проверьте наличие:
   - **graph.pdf** — полный граф (дерево AST + синие рёбра вызовов, цветные узлы по типам).
   - **graph_calls.pdf** — только граф вызовов (FunctionDef и Call, синие связи).
6. Откройте PDF: узлы с подписями имён функций, синие рёбра — вызовы, серые — структура дерева.

---

## 5. Проверка без COMPILE_COMMANDS (опционально)

1. Во временной копии **LOG_PATH.txt** уберите или закомментируйте строку **COMPILE_COMMANDS** (или задайте несуществующий путь).
2. Запустите анализ с **SEARH_METH=CLANG_AST**: должен вывести ошибку о необходимости COMPILE_COMMANDS (validate_config).
3. Для графа: при отсутствии Clang/ccc граф строится с минимальными узлами (пустой Module на файл); **graph.pdf** и **graph_calls.pdf** могут быть почти пустыми, но без падения.

---

## 6. Сборка (PyInstaller / deb)

1. Соберите приложение так, как обычно (например `pyinstaller cpp_analyzer.spec` или сборка deb).
2. В **cpp_analyzer.spec** должен быть включён модуль **graph_runner** (и остальные зависимости).
3. После сборки запустите exe/пакет и повторите шаги 2–4: анализ, результаты в GUI, построение графа.

---

## Краткая последовательность

| № | Действие |
|---|----------|
| 1 | Настроить LOG_PATH.txt (DB, PATH_FILE, SEARH_METH=CLANG_AST, COMPILE_COMMANDS). |
| 2 | Запустить анализ: `_Starter.py` или через GUI. |
| 3 | В GUI → Результаты: проверить таблицы переменных и неиспользуемых (колонки область, критичность и т.д.). |
| 4 | В GUI → Построение графа: указать каталог и сохранить; проверить graph.pdf и graph_calls.pdf. |
| 5 | (По желанию) Проверить поведение без COMPILE_COMMANDS и сборку exe/deb. |

После прохождения пунктов 1–4 все основные нововведения считаются проверенными.
