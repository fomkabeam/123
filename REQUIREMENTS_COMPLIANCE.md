# Соответствие анализатора требованиям РД НДВ (уровни 2–4)

Пошаговый план развития статического контура (AST, отчёт, маршруты): см. **`NDV_AST_ROADMAP.md`**.

Проверка соответствия функциональности проекта требованиям к статическому и динамическому анализу. Контроль состава документации и ручные проверки не рассматриваются.

---

## 1. Четвёртый уровень контроля (3.2)

### 3.2.2. Контроль исходного состояния ПО

| Требование | Реализация | Статус |
|------------|------------|--------|
| Контрольные суммы загрузочных модулей и исходных текстов по каждому файлу | **gui_checksums.py**: КС Deb (MD5/ГОСТ), КС архивов (MD5/ГОСТ). Расчёт КС по файлам. | ✓ Есть |

| Требование | Реализация | Статус |

### 3.2.4. Отчётность

| Требование | Реализация | Статус |

Отдельного протокола «исходники vs объектный код» нет. | ⚠ Частично (КС и избыточность есть; соответствие исходников объектному коду — нет) |

## 2. Третий уровень контроля (3.3)

### 3.3.3. Статический анализ (дополнительно к 4-му уровню)

| Требование | Реализация | Статус |
|------------|------------|--------|
| Контроль полноты и отсутствия избыточности на уровне **функциональных объектов (процедур)** | **createReport** / **gui_results**: неиспользуемые функции (функции без связей), неиспользуемые переменные. **find_func_cpp**: определения и вызовы функций. | ✓ Частично (выявление неиспользуемых процедур/функций) |
| Контроль связей функциональных объектов **по управлению** | **Граф-анализатор**: find_func_cpp — связи Call→FunctionDef; fill_graph — дерево по полю (body, name, func и т.д.). **create_graph_cpp**: полный граф и graph_calls (модули, процедуры, функции и вызовы). | ✓ Есть |
| Контроль связей функциональных объектов **по информации** | **clang_ast_connect**: связи между файлами (#include). **findUseFunction** / **usingVariables**: вызовы и использования. **ccc_connect_list**, **createReport**: файлы/функции/переменные со связями. | ✓ Есть |
| Контроль информационных объектов (локальные, глобальные, внешние переменные и т.п.) | **clang_ast_variables**, **ccc_definition_variable**, **ccc_variables_v**, **ccc_not_use_variable_ast**. **gui_results**: переменные, инициализация, критичность. | ✓ Есть |
| Формирование **перечня маршрутов выполнения** функциональных объектов (процедур, функций) | **Граф вызовов** (create_full_graph, graph_calls) — граф вызовов по сути задаёт маршруты выполнения процедур/функций. Отдельного текстового «перечня маршрутов» в виде списка в коде не формируется; граф и таблицы cpp_func_def/cpp_func_calls можно считать представлением маршрутов. | ✓ Частично (граф = маршруты; явного перечня нет) |

### 3.3.4. Динамический анализ

| Требование | Реализация | Статус |
|------------|------------|--------|
| Контроль выполнения функциональных объектов (процедур, функций) | **Сенсоры** (pasteSensor, NEW_5_JSensor_*): вставка сенсоров в код. **ccc_sensoragramma**: записи срабатываний. **gui_dynamic.py**: статистика по сенсорам. | ✓ Есть |
| Сопоставление фактических маршрутов с маршрутами, построенными при статическом анализе | Сенсораграмма даёт фактические срабатывания; статические маршруты — граф вызовов и ccc_*. Явного автоматического сопоставления «факт vs статика» в одном отчёте не найдено; сопоставление возможно вручную по графу и сенсораграмме. | ⚠ Частично (данные есть; автоматическое сопоставление не реализовано) |

### 3.3.5. Отчётность

Результаты перечисленных проверок частично попадают в отчёт (createReport, gui_results, экспорт CSV). Специального протокола по форме РД с разделами по каждому пункту 3.3.5 в коде нет — оформление отчёта делается вручную.

## 4. Граф-анализатор (кратко)

| Требование РД | Реализация в граф-анализаторе |
|---------------|-------------------------------|
| Связи функциональных объектов по управлению (модули, процедуры, функции) | find_func_cpp (Call→FunctionDef), fill_graph (дерево AST), set_block_id_cpp (блоки), create_full_graph / graph_calls (PDF). |
| Построение блок-схем, диаграмм по исходным текстам | create_full_graph, graph_calls — графы вызовов в .gv и PDF (LR, margin 0.02, красные/синие рёбра). |
| Перечень/представление маршрутов выполнения процедур, функций | Граф вызовов и таблицы cpp_func_def, cpp_func_calls задают маршруты вызовов; явного текстового перечня маршрутов нет. |

Граф-анализатор соответствует требованиям к контролю связей по управлению и к построению диаграмм по исходным текстам. Данные для сопоставления с маршрутами динамического анализа (сенсоры) имеются; автоматическое сопоставление не реализовано.

## 5. Итоговые замечания

- **Реализовано:** контроль исходного состояния (КС), избыточность на уровне файлов, связи по управлению и по информации, информационные объекты (переменные), граф вызовов (диаграммы), маршруты на уровне ветвей (ccc_line_seg), анализ критических маршрутов, динамический анализ (сенсоры), отчётность в виде GUI и отчётов (createReport, CSV).
- **Частично или не реализовано:**
  1. **Соответствие исходных текстов объектному (загрузочному) коду** (3.2.3) — в коде не найдено.
  2. **База потенциально опасных конструкций** и явный синтаксический контроль по ней (3.4.3) — есть вспомогательные скрипты (NEW_SUP_CHECK_POS и др.), но не выделена отдельная база и проверка «по списку».
  3. **Автоматическое сопоставление** фактических маршрутов (процедур/функций и ветвей) с маршрутами статического анализа (3.3.4, 3.4.4) — данные есть, автоматизации нет.
  4. **Сравнение алгоритма по блок-схемам/диаграммам с пояснительной запиской** — только вручную.

Документацию и ручные процедуры (в т.ч. оформление протоколов) вы не проверяли — они не отражены в этом файле.
