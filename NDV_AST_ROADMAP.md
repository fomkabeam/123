# Дорожная карта: приоритет B — качество статического анализа (НДВ / РД)

Документ задаёт **пошаговый план работ** по усилению AST-контура и отчётности под требования **РД по классификации ПО по уровню контроля отсутствия недекларированных возможностей** (уровни **II–IV**, разделы 3.4–3.2), в части **анализа исходных текстов** и **дистрибутивов**. Контроль состава документации, пояснительных записок и ручных процедур испытаний **в анализаторе не реализуется** (как и ранее).

Термины РД (п. 2): **функциональные объекты** (процедуры, функции, ветви…), **информационные объекты** (переменные и др.), **маршруты** выполнения, **статический** и **динамический** анализ.

---

## Связь уровня `CHECK_DEPTH` в GUI с РД

| GUI `CHECK_DEPTH` | Уровень контроля по РД (п. 1.2–1.5) | Акцент требований к статике (кратко) |
|---------------------|-------------------------------------|--------------------------------------|
| 4 | IV (базовый из рассматриваемых) | Файлы, избыточность на уровне файлов, соответствие исходников объектному коду (отчётность). |
| 3 | III | + функциональные объекты (процедуры), связи по управлению и информации, типы информационных объектов, перечни маршрутов; динамика + сопоставление маршрутов. |
| 2 | II | + ветви, база опасных конструкций, маршруты по ветвям, критические маршруты, сопоставление с пояснениями (вне объёма автоматизации документации). |

Реализация в продукте **постепенно приближает** метрики к таблице выше; полное закрытие всех формулировок РД может требовать внешних средств (сборка объектного кода, экспертные списки объектов).

## Этап B1 — Вызовы и связи по управлению (AST)

**Цель:** минимизировать ложные связи и пробелы в графе вызовов относительно требований к **связям функциональных объектов по управлению** (ур. III–IV).

**Чек-лист B1.1 (роли, приёмка, варианты 4.3):** см. **`CHECKLIST_B1_1.md`**. Сначала доработки в коде, **один** приёмочный прогон на реальном проекте — по **`SMOKE_TESTS.md`** (раздел «Приёмочный прогон B1»); от вас не требуется ручной аудит до разработки.

| Шаг | Действие | Критерий готовности |
|-----|----------|----------------------|
| B1.1 | Проверить заполнение `ccc_use_function_list` / аналогов из **clang_ast_runner**; унифицировать с графом (`find_func_cpp`, `graph_runner`). | Один «источник правды» для вызовов при наличии `compile_commands`. |
| B1.2 | Метрики: доля TU с полным разбором, число вызовов без разрешённого callee — вывод в отчёт / `ccc_ast_parse_failures`. | В отчёте видно качество покрытия AST. |
| B1.3 | Экспорт «неоднозначных» или несопоставленных вызовов (опционально в CSV). | Эксперт может разобрать остаток вручную. |

## Этап B2 — Избыточность и «неиспользуемые» сущности (AST)

**Цель:** **контроль полноты и избыточности на уровне функциональных объектов и информационных объектов** (п. 3.3.3, 3.4.3).

| Шаг | Действие | Критерий готовности |
|-----|----------|----------------------|
| B2.1 | Для сценария **CLANG_AST**: неиспользуемые функции/переменные только из AST-представлений (`ccc_not_use_*_ast` и т.д.), без перезаписи regex-легаси без явного режима. | В отчёте и GUI явно видно происхождение данных. |
| B2.2 | Согласовать критерии «не используется» с ограничениями C++ (callback, макросы, условная компиляция) — хотя бы предупреждения в отчёте. | Снижение ложных «мертвых» сущностей. |

## Этап B3 — Диагностика (clang-tidy) и «опасные конструкции» (ур. II)

**Цель:** опереться на **синтаксический контроль наличия заданных конструкций из списка потенциально опасных** (п. 3.4.3).

| Шаг | Действие | Критерий готовности |
|-----|----------|----------------------|
| B3.1 | Зафиксировать набор проверок `CLANG_TIDY_CHECKS` / профиля под НДВ (в т.ч. из вашей практики РД). | Повторяемый прогон на разных проектах. |
| B3.2 | В **createReport** / **gui_results**: блок «Диагностика ↔ требования НДВ» — счётчики по severity, топ файлов, связь с путём к исполняемому сценарию. | В протоколе видна связь статического анализа с отчётом. |
| B3.3 | (Опционально) Таблица соответствия: имя проверки clang-tidy → формулировка требования РД (внутренний справочник, не для пользователя в GUI обязательно). | Упрощение защиты отчёта. |

**Цель:** усилить **перечень маршрутов** и **анализ критических маршрутов** (п. 3.3.3, 3.4.3–3.4.5).

| Шаг | Действие | Критерий готовности |
|-----|----------|----------------------|
| B4.1 | Явный экспорт маршрутов (цепочки вызовов) из графа в текст/CSV для уровня III. | Можно приложить к протоколу. |
| B4.2 | Интеграция с **gui_critical_routes** и списками информационных объектов (куда уже заложено в продукте). | Сценарий «задан экспертом список объектов → критические маршруты». |

## Этап B5 — Сопоставление статики и динамики (после B1–B4)

**Цель:** **сопоставление фактических маршрутов с маршрутами статического анализа** (п. 3.3.4, 3.4.4).

| Шаг | Действие | Критерий готовности |
|-----|----------|----------------------|

| B5.2 | Отчёт о расхождениях (статика предсказала / динамика не подтвердила и наоборот). | Таблица для протокола испытаний. |

## Напоминание: прочие приоритеты (после или параллельно узкими шагами)

| Код | Направление | Когда имеет смысл |
|-----|-------------|-------------------|
| **A** | Оркестратор (`_Starter`), конфиг без глобального `folder`, меньше `os.system` | Стабильность разработки и сопровождения. |
| **C** | Один модуль инструментирования сенсоров, dry-run, backup | Когда динамика — в фокусе защиты. |
| **D** | Статус среды в GUI (БД, compile_commands, libclang, приёмник), прогресс по шагам | Удобство эксплуатации. |
| **E** | Smoke-тесты, CI, воспроизводимая сборка `.deb` | Регрессии при каждом релизе. |

После завершения **B1–B3** разумно коротко вернуться к **A** (оркестратор) или **E** (регрессия), затем **B4–B5** и **C/D** по нагрузке.

## Как пользоваться этим файлом

1. Брать **один этап** (например B1.1), реализовать, зафиксировать в журнале/коммите.  
2. Обновлять таблицы статуса в `REQUIREMENTS_COMPLIANCE.md` при существенном сдвиге.  
3. Не смешивать в одном PR крупный рефакторинг оркестратора (**A**) с логикой AST (**B**), если нет необходимости.
