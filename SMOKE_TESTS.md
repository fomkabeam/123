# Smoke Tests (Analyzer)

Минимальные регрессионные прогоны после изменений пайплайна.

## Подготовка
- Проверить, что в `LOG_PATH.txt` корректны `DB_*`, `PATH_FILE`, `CHECK_DEPTH`, `SEARH_METH`.
- Для AST-прогона подготовить `COMPILE_COMMANDS`.

## Сценарий 1: 4 / ALL (без COMPILE_COMMANDS)
1. Установить:
   - `CHECK_DEPTH=4`
   - `SEARH_METH=ALL`
   - `COMPILE_COMMANDS=` (пусто)
2. Запустить анализ из GUI или:
   - `cpp-analyzer --analyze`
3. Ожидаемо:
   - этапы `find_files`, `findConnectionBetweenFilesCpp`, `createReport`;
   - нет `*_FAILED`;
   - итог GUI: «Анализ завершен успешно».

## Сценарий 2: 3 / ALL (без COMPILE_COMMANDS, legacy fallback)
1. Установить:
   - `CHECK_DEPTH=3`
   - `SEARH_METH=ALL`
   - `COMPILE_COMMANDS=` (пусто)
2. Запустить анализ.
3. Ожидаемо:
   - в логе есть сообщение про legacy fallback;
   - нет попыток AST-шагов, требующих compile_commands;
   - нет `*_FAILED`;
   - итог GUI успешный.

## Сценарий 3: CLANG_AST (с валидным COMPILE_COMMANDS)
1. Установить:
   - `SEARH_METH=CLANG_AST`
   - `COMPILE_COMMANDS=/path/to/compile_commands.json`
2. Запустить анализ.
3. Ожидаемо:
   - выполняются AST/clang шаги пайплайна;
   - при отсутствии обязательного файла анализ падает с понятной ошибкой, не как «успех»;
   - при корректной конфигурации итог успешный.

## Приёмочный прогон B1 (после доработок B1.1 — один раз на реальном проекте)

Цель: не гонять полный цикл на каждой мелкой правке; выполнить **после** накопления изменений по вызовам AST ↔ граф.

**Подготовка:** валидный `COMPILE_COMMANDS`, рабочая БД, `PATH_FILE` на исследуемый проект, `CLANG_LIB` при необходимости.

1. Зафиксировать версию сборки (`cpp-analyzer` / `.deb` + дата).
2. Запустить анализ с **`SEARH_METH=CLANG_AST`** (или полный `ALL` с тем же `compile_commands`, если так принято у вас).
3. В БД (через `psql` или вкладки «Результаты»):
   - есть строки в `ccc_definition_function`, `ccc_use_function_list`;
   - после построения графа (если входит в сценарий) — `cpp_func_*` согласованы с выбранной стратегией B1.1 (не пусто там, где ожидается заполнение из AST).
4. Открыть граф вызовов (если применимо) — PDF/просмотр без падений.
5. Зафиксировать в заметке: числа из п. 3, наличие `*_FAILED` в логе.

**Неуспех:** пустые `ccc_*` при успешном логе; явное расхождение отчёта и графа без пометки источника; необработанный `_FAILED`.

---

## Критерии прохождения
- Любая ошибка шага отражается как `<step>_FAILED: код ...`.
- GUI завершает прогон с ошибкой, если хотя бы один шаг упал.
- Лог не бывает «пустым успешным» — есть либо шаги Start/End, либо явный fail-step.
