# Точность скриптов анализа и улучшение за счёт Clang

## 1. Насколько правильные и точные текущие скрипты

### Общая картина

Основные данные для отчёта (функции, переменные, связи) собираются **не** Clang’ом, а **Python-скриптами на regex и эвристиках**:

| Данные | Скрипт | Метод | Ограничения точности |
|--------|--------|--------|----------------------|
| Определения функций | `findFunction.py` | Regex (модификаторы, тип, имя, границы блоков) | Шаблоны, макросы, сложные типы, вложенные скобки — легко пропуск или ложное срабатывание |
| Вызовы функций | `findUseFunction_V2.py` | Построчный разбор слов + «имя и следующая лексема `(`» | Не различает вызов и упоминание (адрес, шаблонный аргумент, макрос); один и тот же идентификатор в разных контекстах |
| Переменные (объявления) | `NEW_4_Variabels_C.py` | Regex по типам/именам, привязка к функциям/классам | Сложные типы, шаблоны, макросы |
| Использование переменных | `usingVariables.py` | Поиск имени в строке между «разделителями» + уровень скобок `{}` | Ложные срабатывания при частичном совпадении имени (например, префикс другой переменной) |

Итог: скрипты дают **приближённую** картину. Для «нормального» C++ без тяжёлых макросов и шаблонов — часто приемлемо; в сложном коде возможны:

- завышение/занижение числа использований функций;
- пропуск или лишние «неиспользуемые» переменные/функции;
- ошибки в границах функций и привязке переменных к функциям.

Исправление расчёта в `createReport.py` (уникальные функции со связями вместо числа строк в `ccc_use_function_list`) устраняет только ошибку в **отображении** статистики, но не точность **исходных** данных, которые по-прежнему строятся regex/эвристиками.

## 2. Как сейчас используется Clang

В проекте Clang задействован только в **методе CLANG** (и при ALL с заданным `COMPILE_COMMANDS`):

- **Скрипт:** `clang_runner.py`
- **Инструмент:** `clang-tidy` с фиксированным набором проверок:
  - `clang-diagnostic-unused-function`
  - `clang-diagnostic-unused-variable`
  - `clang-analyzer-deadcode.DeadStores`
  - `clang-analyzer-deadcode.UnreachableCode`
- **Результат:** таблица `clang_diagnostics` (файл, строка, столбец, проверка, сообщение).

Таблицы `ccc_definition_function` и `ccc_use_function_list` при методах CLANG/CLANG_AST заполняются из **clang_ast_runner.py** (libclang AST). Для повышения точности: вызовы из другого TU разрешаются по имени; по паре (имя, число аргументов) различаются перегрузки; при нескольких кандидатах все помечаются как используемые (меньше ложных «неиспользуемых»). **Шаблонные функции** (function template / инстанциирования) помечаются в БД (`is_template`) и **не попадают** в представление «неиспользуемые» (ccc_not_use_function), т.к. в header-only и при анализе по одному TU они часто дают ложные срабатывания. В представление добавлены столбцы **line_detect** и **line**. **Неиспользуемые переменные** по AST Clang собираются скриптом **clang_ast_variables.py**: таблицы `ccc_definition_variable`, `ccc_use_variable_list`, представление `ccc_not_use_variable_ast`. В сценарии CLANG он запускается после clang_ast_runner; отчёт и вкладка «Неиспользуемые переменные» в GUI при наличии этих таблиц используют их, иначе — данные из regex-пайплайна (ccc_variables_v, ccc_unuseVar). Диагностика clang-tidy (unused, DeadStores, UnreachableCode и др.) пишется в `clang_diagnostics` из `clang_runner.py`.

## 3. Как можно улучшить качество с помощью Clang

Есть два основных направления.

### Вариант A: Перекрёстная проверка (минимальные изменения)

Оставить текущий пайплайн как есть, но **сопоставлять** его вывод с результатами Clang:

- В отчёте или в GUI показывать пометки вида: «по нашему анализу не используется» / «подтверждено Clang (unused-function)» / «Clang не сообщает о неиспользуемости».
- Для переменных — аналогично с `clang-diagnostic-unused-variable` и `DeadStores`.

Плюсы: не трогаем парсеры, быстро внедрить. Минусы: точность по-прежнему ограничена нашим парсером; Clang может не разобрать часть файлов (нет include’ов и т.д.).

### Вариант B: Частичная или полная замена парсинга на Clang AST

Использовать **AST Clang** для извлечения фактов:

- **Определения функций** — обход AST (например, через libclang Python bindings или парсинг вывода `clang -Xclang -ast-dump`), запись в те же или новые таблицы.
- **Вызовы функций** — по AST однозначно: вызов функции vs упоминание имени в другом контексте.
- **Переменные и использования** — объявления и использования переменных из AST дают точную привязку к области видимости.

Тогда:

- Таблицы `ccc_definition_function`, `ccc_use_function_list`, `ccc_variables_v` можно заполнять из Clang (при наличии `compile_commands.json`), а текущие скрипты оставить как fallback для проектов без compile_commands или для «быстрого» режима без Clang.
- Либо ввести отдельный «режим CLANG_AST» (или расширить метод CLANG), который заполняет эти таблицы из AST и строит отчёт уже по ним.

Технически это означает:

- Вызов Clang для каждого файла с теми же флагами, что в `compile_commands.json` (как уже делает `clang_runner.py`).
- Парсинг AST через:
  - **libclang** (Python: пакет `libclang` или `clang`), или
  - запуск `clang -Xclang -ast-dump -fsyntax-only` и разбор текстового дампа (менее удобно, но без лишних зависимостей).

Плюсы: семантически корректный разбор, учёт типов, шаблонов, макросов (после препроцессора). Минусы: зависимость от наличия `compile_commands.json` и от того, что Clang успешно разбирает файл (те же ограничения, что и у текущего clang-tidy).

## 4. Практические шаги (если выбирать Вариант B)

1. **Прототип на libclang (один файл):**  
   Написать скрипт, который по одному `.cpp` и его флагам из `compile_commands.json` строит AST, выводит список определений функций и вызовов (имя, строка, столбец). Сравнить с выводом `findFunction` / `findUseFunction_V2` на том же файле.

2. **Интеграция в пайплайн:**  
   Добавить режим (например, `SEARH_METH=CLANG_AST` или опцию в `LOG_PATH.txt`: `USE_CLANG_AST=1` при `SEARH_METH=ALL`), при котором:
   - для каждого файла из compile_commands вызывается Clang (AST),
   - результаты пишутся в `ccc_definition_function`, `ccc_use_function_list`, `ccc_variables_v` (схему при необходимости расширить),
   - `createReport` и остальные шаги работают как раньше.

3. **Fallback:**  
   Если для файла нет записи в compile_commands или Clang выдал ошибку разбора — использовать текущие Python-скрипты для этого файла (или помечать «без данных Clang»).

4. **Переменные:**  
   Аналогично: обход AST для объявлений переменных и их использований, запись в `ccc_variables_v` и при желании сопоставление с `ccc_unuseVar` / отчётом.

## 5. Краткий вывод

- **Точность текущих скриптов** — приближённая; в сложном C++ возможны ошибки из-за regex и эвристик.
- **Clang в проекте** сейчас используется только для **диагностики** (clang-tidy → `clang_diagnostics`), а не для заполнения таблиц функций/переменных.
- **Улучшить качество** можно:
  - **Быстро:** перекрёстная проверка нашего отчёта с `clang_diagnostics` (Вариант A).
  - **Существенно:** вынести сбор фактов (функции, вызовы, переменные) на Clang AST и при необходимости оставить текущие скрипты как запасной вариант (Вариант B).

Если нужно, можно набросать минимальный пример на libclang для одного файла (определения функций + вызовы) под вашу структуру БД.
