# Руководство по установке и настройке C++ анализатора

Единый документ для развёртывания анализатора и устранения типичных проблем. Для новичков: выполняйте шаги по порядку.

---

## 1. Зависимости (что должно быть установлено)

| Компонент | Назначение |
|-----------|------------|
| **Python 3.5** | Запуск GUI и скриптов (Astra Linux часто по умолчанию 3.5). |
| **PostgreSQL** | База данных для результатов анализа (таблицы файлов, функций, диагностик и т.д.). |
| **clang, clang-tidy** | Методы CLANG / CLANG_AST: диагностика кода (unused, dead code и т.д.). |
| **LLVM + libclang + привязки Python** | Методы CLANG_AST, граф вызовов, диагностика Clang AST: парсинг C++ через libclang. |

**Для новичков:** без PostgreSQL и Python анализатор не запустится. Без LLVM/libclang не будут работать «Анализ через Clang AST», «Построение графа вызовов» и пункт меню «Диагностика Clang».

---

## 2. Установка базовых компонентов (Astra Linux)

### 2.1 PostgreSQL и Python

```bash
sudo apt-get update
sudo apt-get install postgresql python3
```

Создайте базу и пользователя (если ещё не созданы). Параметры подключения потом укажете в настройках анализатора (файл `LOG_PATH.txt` или окно «Настройки анализатора» в GUI).

### 2.2 Clang и clang-tidy (для метода CLANG)

```bash
sudo apt-get install clang clang-tidy clang-tools
clang --version
clang-tidy --version
```

Нужны для запуска анализа с методом **CLANG** (диагностика в таблицу `clang_diagnostics`). Подробнее — в **README_CLANG.md**.

---

## 3. LLVM и libclang (для AST и графа вызовов) — важно для новичков

Методы **CLANG_AST**, **CLANG_AST_NO_SENS**, построение **графа вызовов** и пункт меню **«Диагностика Clang (парсинг AST)»** используют библиотеку **libclang** и её **Python-привязки**. Версия привязок и версия установленной библиотеки должны совпадать.

На Astra Linux в репозиториях часто есть только **LLVM 6**; пакет **libclang** с pip (PyPI) рассчитан на более новые версии, в PyPI **нет** libclang 6.x — из-за этого возникает ошибка вида:

`undefined symbol: clang_CXXMethod_isDeleted` или «python bindings are compatible with your libclang.so version».

**Рекомендуемый вариант (проверенный):** установить **LLVM 16** и системные привязки **python3-clang-16**.

### 3.1 Проверить, есть ли в репозитории пакеты LLVM 16

```bash
apt-cache search python3 clang
```

Если в списке есть **python3-clang-16** (или python3-clang-19) — переходите к шагу 3.2.

### 3.2 Установить LLVM 16 и привязки

```bash
sudo apt install llvm-16 libclang-16-dev python3-clang-16
```

(Если в репозитории нет `llvm-16`, но есть только `python3-clang-16`, установите его — зависимости подтянут нужную библиотеку.)

### 3.3 Привязки Python: системные или pip (важно)

Есть два рабочих варианта. Если после шага 3.2 при запуске диагностики или анализа появляется **«No module named 'clang'»**, значит скрипт запускается с тем Python, у которого нет системного пакета (например, другой интерпретатор или окружение). Тогда используйте **вариант Б**.

**Вариант А — только системные привязки** (если скрипты всегда запускаются тем же `python3`, для которого установлен python3-clang-16):

```bash
pip3 uninstall libclang
```

На вопрос об удалении зависимостей можно ответить «y». Дальше — шаг 3.4.

**Вариант Б — pip libclang 16 + системная библиотека** (рекомендуется, если уже появилась ошибка «No module named 'clang'»):

Установить привязки с pip **той же версии**, что и LLVM 16, и указать путь к **системной** libclang — тогда загружается нужная библиотека и модуль `clang` есть:

```bash
pip3 install 'libclang==16.0.6'
```

В **LOG_PATH.txt** (или в настройках GUI) обязательно задать путь к библиотеке LLVM 16 (шаг 3.4). Тогда при импорте будет использоваться системная `/usr/lib/llvm-16/lib/libclang.so.1`, а не та, что идёт в комплекте с pip-пакетом. Дальше — шаг 3.4.

### 3.4 Указать путь к библиотеке libclang

Путь к библиотеке задаётся в конфигурации (файл **LOG_PATH.txt** или окно **«Настройки анализатора»** в GUI), ключ **CLANG_LIB**.

Типичные пути после установки LLVM 16:

- `/usr/lib/llvm-16/lib/libclang.so.1`
- или `/usr/lib/llvm-16/lib/libclang-16.so.1`

Проверить наличие файла:

```bash
ls /usr/lib/llvm-16/lib/libclang*.so.1
```

**В GUI:** в разделе «Пути и файлы» есть поле **«Библиотека libclang (для AST и графа вызовов)»**:

- можно ввести путь вручную;
- кнопка **«Найти»** — подставляет первый найденный путь в стандартных каталогах (`/usr/lib/llvm-16/`, `llvm-19`, `llvm-14`, `llvm-6.0`);
- кнопка **«Обзор...»** — выбор файла вручную.

При первом открытии настроек, если **CLANG_LIB** в конфиге пустой и libclang найден в стандартном пути, значение подставится автоматически. Не забудьте нажать **«Сохранить»**.

В **LOG_PATH.txt** строка будет выглядеть так:

```text
CLANG_LIB=/usr/lib/llvm-16/lib/libclang.so.1
```

## 4. Файл конфигурации LOG_PATH.txt

Файл **LOG_PATH.txt** лежит рядом с исполняемым файлом анализатора (или в корне проекта «С++»). Формат: одна строка на параметр, `КЛЮЧ=значение`. Комментарии — строки, начинающиеся с `#`.

### 4.1 Основные ключи (обязательные для работы)

| Ключ | Описание | Где задать в GUI |
|------|----------|------------------|
| DB_NAME | Имя базы PostgreSQL | Настройки анализатора → Настройки БД |
| DB_USER | Пользователь БД | то же |
| DB_PASS | Пароль БД | то же |
| DB_HOST | Хост БД | то же |
| DB_PORT | Порт БД | то же |
| PATH_FILE | Путь к каталогу проекта C++ | Пути и файлы → Путь к проекту |
| SEARH_METH | Метод анализа (ALL, CLANG, CLANG_AST и др.) | Параметры анализа → Метод поиска |
| CHECK_DEPTH | Уровень контроля (2, 3, 4) | Параметры анализа → Уровень контроля |

### 4.2 Ключи для Clang / AST / графа

| Ключ | Описание | Где в GUI |
|------|----------|-----------|
| COMPILE_COMMANDS | Полный путь к `compile_commands.json` (каталог build после CMake) | Пути и файлы → Файл команд компиляции |
| CLANG_LIB | Путь к libclang (например `/usr/lib/llvm-16/lib/libclang.so.1`) | Пути и файлы → Библиотека libclang (кнопки «Найти», «Обзор...») |
| CLANG_TIDY_CHECKS | Доп. проверки clang-tidy через запятую | Пути и файлы → Доп. проверки clang-tidy |

Подробнее про clang-tidy и рекомендуемые проверки — **README_CLANG.md**.

## 5. Первый запуск (кратко)

1. Установить зависимости (разделы 2 и 3).
2. Создать БД PostgreSQL, запомнить имя, пользователя, пароль, хост, порт.
3. Запустить анализатор (GUI).
4. Открыть **«Настройки анализатора»**, заполнить параметры БД, путь к проекту, при необходимости — COMPILE_COMMANDS и CLANG_LIB (кнопка «Найти» для libclang), нажать **«Сохранить»**.
5. Выбрать метод анализа (например CLANG_AST при наличии COMPILE_COMMANDS и CLANG_LIB).
6. Запустить анализ.

## 6. Устранение неполадок

### Ошибка «No module named 'clang'» при запуске диагностики или анализа

**Причина:** модуль `clang` не найден. Так бывает, если удалили pip-пакет `libclang`, а скрипт запускается с тем Python, для которого не установлен системный пакет `python3-clang-16` (другой интерпретатор или окружение).

**Решение:** установить привязки с pip той же версии, что и LLVM 16, и указать системную библиотеку в CLANG_LIB:

```bash
pip3 install 'libclang==16.0.6'
```

В **LOG_PATH.txt** или в настройках GUI задать (см. раздел 3.4):

```text
CLANG_LIB=/usr/lib/llvm-16/lib/libclang.so.1
```

После этого модуль `clang` будет доступен, а загружаться будет системная libclang 16.

### Ошибка «undefined symbol: clang_CXXMethod_isDeleted» / «python bindings are compatible with your libclang.so version»

**Причина:** несовпадение версии библиотеки libclang и Python-привязок (часто на Astra — системный LLVM 6 и новый пакет libclang с pip).

**Решение:** установить LLVM 16 (раздел 3.2), затем либо системные привязки (3.3 вариант А), либо `pip3 install 'libclang==16.0.6'` и обязательно указать CLANG_LIB (раздел 3.4).

### Ошибка «No module named 'dbFolder'» при запуске диагностики Clang

**Причина:** скрипт диагностики запускается из каталога, где нет модулей проекта (например из `/opt/cpp-analyzer` после установки .deb).

**Решение:** уже учтено в скрипте: конфиг читается из **LOG_PATH.txt** в каталоге скрипта, текущем каталоге или `/opt/cpp-analyzer`. Убедитесь, что **LOG_PATH.txt** лежит в одном из этих мест и в нём заданы COMPILE_COMMANDS и при необходимости CLANG_LIB.

### Ошибка «module 'os' has no attribute 'fspath'»

**Причина:** Python 3.5 (в 3.6 появился `os.fspath`). В диагностическом скрипте добавлен обходной вариант для 3.5; если ошибка остаётся — проверьте, что запускается именно этот скрипт из поставки.

### Пустой AST / «не удалось разобрать файл» при построении графа

**Причина:** Clang не находит заголовки или макросы (нужны правильные пути `-I` и флаги из compile_commands.json).

**Решение:** в логе построения графа после последних правок выводятся **конкретные сообщения Clang** (файл не найден, необъявленный идентификатор). По ним добавьте пути в настройки сборки или проверьте путь к **COMPILE_COMMANDS**. Подробнее — в **README_CLANG.md** и в выводе «Диагностика Clang» из меню.

### Построение графа: «Сохранение графа в PDF» висит много часов

**Причина:** после записи полного графа запускается утилита **dot** (Graphviz) для раскладки и генерации PDF. При очень большом графе (сотни тысяч рёбер) dot может работать многие часы или не завершиться.

**Решение:** при **большом графе** (более 80 000 узлов или 250 000 рёбер) программа **предлагает выбор** в диалоге:
- сохранять ли полный исходный граф в **graph_full_original.gv** (все узлы и рёбра);
- что делать с PDF: только граф вызовов (**graph_calls.pdf**) — быстро, полный граф в PDF — может занять часы, или не строить PDF.

Автоматического принудительного пропуска нет: пользователь сам выбирает. Если текущий запуск уже долго висит на этапе «Сохранение графа в PDF», можно прервать процесс (Остановить) и при следующем запуске в диалоге выбрать, например, только сохранение .gv и граф вызовов.
